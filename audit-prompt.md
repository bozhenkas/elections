## 0. Правила по комментариям

НЕ удалять:
- комментарии с пометкой TODO, FIXME, HACK, NOTE
- комментарии начинающиеся с "// dev:", "// backend:", "// api:"
- JSDoc блоки в файлах src/api/
- любые комментарии явно адресованные другим разработчикам (объясняющие почему что-то сделано именно так, предупреждения, контракты)

УДАЛЯТЬ:
- закомментированный код
- очевидные комментарии типа "// импорт компонента", "// стейт для модалки", "// рендер списка"
- console.log, console.error, debugger
- автосгенерированные комментарии от ИИ ("// this function handles...", "// here we...")

ДОБАВИТЬ:
- короткую документацию для сложных мест: нетривиальная логика, неочевидные решения, хитрые вычисления
- формат: комментарий на русском, с маленькой буквы, сухо, без воды
- пример: // пересчёт процентов после каждого голоса, округление до десятых
- пример: // debounce 300ms чтобы не слать повторные запросы при быстрых кликах
- пример: // фокус-трап внутри модалки: tab циклится между первым и последним элементом
- НЕ писать документацию к простым вещам. если код читается без комментария — не пиши

## 1. Структура проекта

Проверь и при необходимости реорганизуй файловую структуру:
- Компоненты в src/components/, каждый в своей папке (Component/index.jsx + Component.module.css)
- Хуки в src/hooks/
- API-заглушки в src/api/
- Моковые данные в src/mock/
- Стили-токены в src/styles/tokens.css
- Локали в src/locales/
- Утилиты в src/utils/
Удали неиспользуемые файлы, дублирующиеся стили, пустые папки.

## 2. Clean code

Для КАЖДОГО файла в проекте:
- Удали неиспользуемые импорты, переменные, функции, пропсы
- Переименуй нечитаемые переменные (e, d, tmp, data2) в понятные
- Разбей функции длиннее 30 строк на подфункции
- Вынеси magic numbers в константы (таймауты, размеры, лимиты)
- Замени вложенные тернарники на early return или отдельные компоненты
- Убедись что каждый компонент делает одну вещь

## 3. React best practices

- Каждый компонент — функциональный (никаких class components)
- Пропсы деструктурированы в сигнатуре функции
- Все списки имеют стабильный key (не index)
- useEffect имеет корректные зависимости, cleanup где нужно
- Нет утечек состояния (setState после unmount)
- Нет лишних ре-рендеров: memo, useMemo, useCallback — только там где реально нужно, не везде подряд
- Стейт поднят на минимально необходимый уровень
- Нет prop drilling глубже 2 уровней — использовать context

## 4. CSS / стили

- Убери дублирование: повторяющиеся цвета, шрифты, отступы вынеси в CSS-переменные в tokens.css
- Убери !important
- Убери неиспользуемые классы
- Проверь что нет inline-стилей которые должны быть в CSS
- Проверь адаптив: все breakpoints консистентны
- Убедись что шрифты Serpantin и Decor используются только там где положено по дизайн-системе

## 5. Безопасность

- Проверь все места где рендерится пользовательский ввод — нет ли dangerouslySetInnerHTML без санитизации
- Проверь что токен авторизации хранится в httpOnly cookie или secure storage, НЕ в localStorage
- Проверь что API-заглушки не содержат захардкоженных токенов или паролей
- Проверь что формы защищены от повторной отправки (double submit)
- Проверь что голосование невозможно без авторизации на уровне UI
- Добавь rate limiting на кнопку голосования (disable после клика)
- Убедись что нет открытых редиректов

## 6. Доступность (a11y)

- Все интерактивные элементы доступны с клавиатуры (tab, enter, escape для модалки)
- Модальное окно: focus trap, закрытие по Escape, aria-modal="true"
- Изображения имеют alt
- Кнопки имеют понятный текст или aria-label
- Контраст текста на тёмном фоне (#CDDCDC на #0F0F0F и #243434) соответствует WCAG AA

## 7. Производительность

- Изображения кандидатов — lazy loading
- Шрифты подключены с font-display: swap
- Нет блокирующих запросов в рендере
- Компоненты за модалкой не рендерятся пока она не открыта (lazy mount)
